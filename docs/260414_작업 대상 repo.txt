이번 작업의 핵심 목표:
기존 표장 유사성 판단 로직(발음/호칭, 외관, 관념, 문자열 유사도)은 최대한 유지하되,
대한민국 상표심사 실무에 맞게

1. 니스분류 기반 상품 입력,
2. 유사군코드 중심의 상품 유사성 판단,
3. 상품류/상품군 범위 안에서의 권리범위 판단,
4. 상품-서비스업 간 예외 판단,
5. 선행상표 상태(등록/출원/거절/포기/소멸 등) 반영
   을 추가/보강하라.

이 프로젝트의 새로운 기본 원칙:

* 사용자의 상품 선택 입력 source of truth는 니스분류다.
* 제품(goods)은 제1류~제34류, 서비스(services)는 제35류~제45류다.
* 분류 1 / 분류 2 / 상품군은 UX 레이어이고, 내부 판단은 실제 니스류 번호와 상품군 문맥 기준으로 한다.
* 지정상품의 동일·유사 여부는 상품류 번호만이 아니라 원칙적으로 유사군코드를 기준으로 판단한다.
* 표장 유사성은 기존 로직을 유지하되, 상품 유사성 필터를 통과한 선행상표에 대해 강하게 적용한다.
* 상표의 권리범위 판단은 사용자가 선택한 지정상품 범위(상품류 및 상품군) 안에서 우선 검토한다.

반드시 반영할 법리/심사기준:

1. 유사상품 심사기준:

   * 지정상품의 동일/유사 여부는 상품류 구분과 관계없이 원칙적으로 유사군코드를 기준으로 판단
   * 상품과 서비스업 간은 원칙적으로 유사도가 낮음
   * 다만 출처 혼동 우려가 있으면 예외적으로 유사하게 판단
   * G390802(소프트웨어)와 서비스업 간은 별도 예외 기준으로 판단
2. 상표심사기준:

   * 식별력 판단과 선등록상표 충돌 판단은 분리된 심사 축으로 처리
   * 지정상품의 기재 및 지정상품 심사 구조를 반영
   * 선등록상표와 동일·유사한 상표 판단은 지정상품의 동일·유사 여부와 함께 보아야 함
3. 니스분류:

   * 상품은 1~34류
   * 서비스는 35~45류
   * 사용자는 제품/서비스 선택 후 대분류와 상품군을 선택해야 함

절대 유지할 것:

* 기존 발음/호칭 유사도 판단 로직
* 기존 외관/관념/문자열 유사도 로직
* 기존 KIPRIS 검색 흐름
* 기존 유사군코드 추천/매핑 로직
* 기존 step 기반 UI의 큰 흐름
* 기존 식별력 평가 로직의 기본 축

전면 수정이 아니라, 아래 구조를 덧붙여 판단 순서를 바로잡아라.

새 기본 판단 구조:

1. 입력 정규화
2. 식별력 판단
3. 지정상품 범위 확정
4. 상품 유사성 필터
5. 표장 유사성 판단
6. 선행상표 상태 반영
7. 혼동 가능성 종합
8. 최종 점수/설명 생성

A. 입력 정규화 구조
사용자 선택값을 아래처럼 정규화하라.

* selected_kind: goods or services
* selected_groups: UI 대분류 목록
* selected_subgroups: 구체 상품군 목록
* selected_nice_classes: 실제 니스류 번호 목록
* selected_similarity_codes: 유사군코드 목록
* selected_keywords: 상품군 문맥 키워드
* specific_product_text: 사용자가 직접 입력한 상품명(있으면)

이 값들이 이후 모든 판단의 시작점이 되게 하라.

B. 니스분류 기반 지정상품 범위 확정

1. 사용자가 제품을 선택하면 goods(1~34류)만 선택 가능
2. 사용자가 서비스를 선택하면 services(35~45류)만 선택 가능
3. 대분류와 상품군은 UX용 묶음이지만, 내부적으로는 반드시 실제 니스류로 귀속
4. 복수 상품군 선택 시 연결된 니스류는 dedupe
5. 이후 선행상표 비교 시 selected_nice_classes와 selected_similarity_codes를 모두 사용

C. 상품 유사성 필터를 가장 먼저 적용
선행상표 후보는 아래 4개 그룹으로 분리하라.

1. exact_scope_candidates

* 동일 유사군코드
* 동일 또는 강하게 연결된 상품군
* 가장 우선 검토

2. same_class_candidates

* 동일 니스류이나 유사군코드는 다름
* 보조 검토군

3. related_market_candidates

* 타 류 또는 상품-서비스업 사이지만
* 출처 혼동 가능성이 있는 경우
* 예외 검토군

4. irrelevant_candidates

* 타 류, 타 코드, 상품군 관련성 없음
* 최종 점수 직접 반영 금지

중요:

* 최종 점수 감점은 exact_scope_candidates와 일부 same_class_candidates 중심으로 하라
* irrelevant_candidates는 검색 결과에는 보여줄 수 있지만 점수에는 직접 반영하지 마라

D. 상품-서비스업 간 유사 판단 예외
유사상품 심사기준을 반영해 아래 로직을 넣어라.

1. 상품과 서비스업 간은 원칙적으로 유사도 낮음
2. 다만 아래를 종합 고려해 예외 검토

* 동일 사업자가 통상 제공하는지
* 상품의 용도와 서비스 내용이 일치하는지
* 판매장소/제공장소가 일치하는지
* 수요자 범위가 일치하는지
* 양 표장이 전체적으로 동일 또는 극히 유사한지

3. 특히 소프트웨어(G390802)와 서비스업은 별도 예외 로직

* 제42류(소프트웨어 개발, SaaS 등)
* 제35류 일부 플랫폼/판매 관련 서비스
* 제38류 일부 통신/플랫폼 서비스
  와의 관계를 개별적으로 검토
* 단, 양 표장이 전체적으로 동일 또는 극히 유사한 경우에 한하여 강한 유사 판단

E. 기존 표장 유사성 로직은 유지하되 적용 대상을 조정
현재 발음/호칭/외관/관념/문자열 유사도 로직은 유지한다.
다만 아래처럼 바꿔라.

1. 상품 유사성 필터를 통과한 후보에 대해서는 기존 로직을 강하게 적용
2. 필터를 통과하지 못한 후보에 대해서는 참고용 계산만 가능하지만 최종 감점은 금지
3. 완전 동일 표장(exact identity)은 mark_identity = exact 로 분류
4. exact + 동일 유사군코드 + 등록/출원 상태면 최고 위험군
5. exact라도 irrelevant_candidates면 직접 감점하지 마라

F. 선행상표 상태 반영
선행상표를 상태별로 아래처럼 나누어라.

1. live_blockers

* 등록
* 출원
* 공고
* 심사중
  -> 최종 점수에 직접 반영

2. historical_references

* 거절
* 포기
* 취하
* 소멸
* 말소
  -> 원칙적으로 최종 점수 직접 반영 금지
  -> 참고자료 또는 보조 경고로만 사용

G. 거절 상표 처리 로직
거절된 상표는 자동으로 위험후보로 보지 말고 아래처럼 처리하라.

1. 기본 원칙

* 최종 점수 직접 감점 금지
* top_prior 우선순위 낮춤

2. 예외

* 거절결정문/행정정보가 확보된 경우

  * cited_marks
  * refusal_core
  * refusal_basis
  * weak_elements
  * current_mark_relevance
    를 저장하고,
    현재 검토상표와 직접 관련성이 높을 때만 보조 경고로 반영

3. 예시

* FlexAiCam 거절은 AiCam/에이캠 계열과의 충돌 사례라면
  LexAI 평가에는 직접 감점하지 않고 역사적 참고자료로 처리

H. 권리범위 반영 원칙
이 프로젝트의 권리범위 판단은 아래를 기준으로 한다.

1. 등록 또는 출원 상표의 보호/충돌 범위는

* 사용자가 선택한 지정상품의 니스류
* 선택한 상품군 문맥
* 선택된 유사군코드
  안에서 우선적으로 미친다고 본다.

2. 동일 표장이라도

* 상품류/상품군/유사군코드가 실질적으로 멀면
  원칙적으로 직접 충돌로 보지 않는다.

3. 동일/유사한 상품군 범위 안에 있는 경우

* 표장 유사성 판단을 강하게 반영한다.

4. 동일 유사군코드가 가장 강한 기준이고,
   니스류는 보조적 구조화 기준으로 사용한다.

I. 최종 점수 캘리브레이션
최종 점수는 아래 3축으로 합성하라.

* 식별력 리스크
* 상품 유사성/권리범위 리스크
* live_blockers 기준 선행상표 충돌 리스크

반드시 아래 원칙을 지켜라.

1. 상품 유사성 필터를 통과한 선행상표가 없으면
   상대적 거절사유 리스크는 매우 낮게 본다
2. 동일 유사군코드 + 높은 표장 유사도 + 등록/출원 상태면
   강한 위험으로 본다
3. 거절/포기/취하/소멸 상표는 직접 감점하지 않는다
4. 타 류/타 상품군/타 코드 후보만으로 점수가 과도하게 낮아지지 않게 한다

L. 테스트 케이스 추가
반드시 아래 테스트를 포함하라.

1. 동일 표장 + 동일 유사군코드 + 등록 상태

* 최고 위험군

2. 동일 표장 + 타 류 + 무관한 상품군

* 직접 감점하지 않음

3. 동일 니스류 + 다른 유사군코드

* 중간 위험군 또는 보조 검토군

4. 소프트웨어(G390802)와 제42류 서비스업

* 예외 검토 적용

5. 거절 상표만 존재하는 경우

* 직접 감점 금지

6. 사용자가 제품/서비스를 선택하고,
   대분류/상품군을 고르면 실제 니스류와 유사군코드가 저장되는지 확인

1. 수정 코드
2. 수정 문서
3. 변경 요약

5. 테스트 결과
6. 실제 예시 설명

   * 제품 20류 가구
   * 9류 소프트웨어
   * 서비스 41/42류
   * 거절상표 보조 참고 사례
