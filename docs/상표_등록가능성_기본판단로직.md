# 상표 등록가능성 기본판단로직

## 1. 전체 구조
- 최종 등록가능성은 `min(Stage 1 absolute cap, Stage 2 relative score)`로 계산한다.
- Stage 1은 절대적 거절사유다.
- Stage 2는 선행상표와의 상대적 거절사유다.
- Stage 2에서는 니스류보다 item-level 유사군코드(SC)를 우선한다.

## 2. KIPRIS 검색 원칙
- 검색은 recall 단계다.
- 1차 검색은 `TN + class + primary SC`를 우선 사용한다.
- 2차 검색은 `TN + class only`, `TN + related SC`, `TN + retail/service SC`를 사용한다.
- 마지막 보조 검색으로 `TN` broad fallback을 사용할 수 있다.
- 검색 단계에서 넓게 찾더라도, 최종 평가는 검색식에 사용한 SC가 아니라 prior designated item에 실제 붙은 item-level SC 기준으로 한다.

## 3. prior mark 상세정보 저장 원칙
선행상표는 mark-level 요약이 아니라 designated item 단위로 저장한다.

각 prior designated item은 최소 아래 필드를 가진다.
- `prior_item_label`
- `prior_class_no`
- `prior_similarity_codes`
- `prior_item_type`
- `prior_underlying_goods_codes`
- `source_page_or_source_field`
- `parsing_confidence`

## 4. item-level overlap_type
- `exact_primary_overlap`
  - selected primary code와 prior item primary code가 직접 일치
  - 예: `S0201 vs S0201`
- `related_primary_overlap`
  - direct는 아니지만 근접 코드군 또는 base goods relation이 확인됨
  - 예: `S0201 vs S120401`
- `retail_overlap_only`
  - 판매업 코드만 일치
  - 자동 유사 금지
- `same_class_only`
  - 같은 니스류일 뿐 direct/related item-level SC overlap 없음
- `no_material_overlap`
  - 실질 중첩 없음

## 5. 점수 캡 규칙
- 등록 상태 prior + high mark similarity + `exact_primary_overlap`
  - Stage 2 upper cap `<= 45`
  - 동일 또는 매우 유사 표장이면 `<= 35`까지 가능
- 등록 상태 prior + high mark similarity + `related_primary_overlap`
  - Stage 2 upper cap `<= 50`
- 등록 상태 prior + high mark similarity + `same_class_only`
  - `60~75`
- `exact_primary_overlap` 또는 `related_primary_overlap`이 1건이라도 있으면 same-class-only 밴드로 재보정하지 않는다.
- direct overlap은 식별력 보정보다 우선한다.

## 6. 금융/보험/부동산/법무 기준
- 금융 `S0201`
  - prior item에 `S0201` 또는 `S120401`이 있으면 강한 충돌로 본다.
  - same-class-only로 떨어지면 버그다.
- 보험 `S0301`
  - prior item에 `S0301` 직접 항목이 없으면 same-class-only 또는 약한 보조 검토군이다.
- 부동산 `S1212`
  - prior item에 `S1212` 직접 항목이 없으면 same-class-only다.
- 법무 `S120402`
  - prior 36류 금융 item과 직접 overlap이 없으면 가장 유리한 축으로 본다.

기대 순서는 대체로 아래다.
- 법무 > 보험/부동산 > 금융

## 7. 판매업 코드 처리
- 상품 유사군코드와 판매업 유사군코드는 별도로 저장한다.
- 판매업 코드가 같다는 이유만으로 direct conflict로 올리지 않는다.
- 반드시 underlying goods relation을 추가 확인한다.
- underlying goods overlap이 없으면 `retail_overlap_only` 약신호로만 남긴다.

예시:
- 귀걸이 `G4509`
- 판매업 `S2045`
- 속옷 판매업도 `S2045`일 수 있다.
- 하지만 귀걸이와 속옷은 자동 유사가 아니다.

## 8. Explainability 출력
보고서에는 최소 아래를 표시한다.
- `selected primary codes`
- `strongest prior item`
- `strongest prior codes`
- `overlap_type`
- `overlap_confidence`
- `cap_reason`
- `stage2_cap_upper`

예시:
- 금융
  - `selected primary codes: S0201`
  - `strongest prior item: 금융 또는 재무에 관한 정보제공업`
  - `strongest prior codes: S0201,S120401`
  - `overlap_type: exact_primary_overlap`
  - `cap_reason: registered prior + high mark similarity + direct code overlap`
  - `stage2_cap_upper: 45`
- 부동산
  - `selected primary codes: S1212`
  - `strongest prior item: 없음`
  - `overlap_type: same_class_only`
  - `stage2_cap_upper: 75`

## 9. 금지사항
- 금융을 same-class-only로 계속 처리하지 않는다.
- direct overlap인데 등록가능성이 75로 유지되면 실패다.
- 설명 문구만 바꾸고 scoring 구조를 그대로 두지 않는다.
- 사용자가 유사군코드를 직접 선택하는 UI로 되돌리지 않는다.

## 10. KIPRIS 검색 안전장치

### 10.1 TN+class fallback은 항상 존재한다
- `build_kipris_search_plan`은 `selected_primary_codes`가 비어 있어도 반드시 아래 두 단계를 포함한다:
  - `class_only`: `TN=[name] AND CLASS=[no]` — SC 없이 류만 사용
  - `text_fallback`: `TN=[name]` — 류 없이 이름만 사용
- primary_codes가 None이거나 []이면 SC 기반 단계만 건너뛰고, class_only는 항상 실행한다.

### 10.2 SC는 검색 강화 도구이지 전제조건이 아니다
- SC를 못 얻어도 검색은 계속 진행된다.
- SC 기반 검색(primary_sc, related_sc, retail_sc)은 정밀도를 높이는 추가 단계다.
- SC 매핑 실패 사유는 `mapping_failed_reason`에 기록하되, 검색을 중단하지 않는다.

### 10.3 "0건 → 90%" 버그 방지 원칙
- 검색이 0건을 반환했다고 해서 자동으로 고득점(90%)을 주면 안 된다.
- prior_items 주입(fixture/API 결과)이 비어 있으면 filtered_prior_count=0이므로 same-class-only 또는 no_material_overlap으로 처리된다.
- 검색 0건의 경우 점수는 60~75 범위에 머물러야 하며, 85 이상은 name distinctiveness가 exceptionally strong한 경우에만 허용한다.

## 11. 상세정보 파싱 → overlap 평가 흐름
1. KIPRIS 검색 결과 → `enrich_search_results_with_item_details`
2. prior designated items 파싱 → `prior_similarity_codes` 추출
3. `normalize_selected_input` → `selected_primary_codes`, `selected_related_codes`, `selected_retail_codes`
4. 각 prior item × selected codes 쌍 → `overlap_type` 결정
5. 가장 강한 overlap_type으로 Stage 2 score cap 결정
6. `min(Stage 1 cap, Stage 2 score)` → 최종 점수

### `selected_primary_codes`가 비어 있어도 흐름은 중단되지 않는다
- `selected_primary_codes=[]` 상태에서는 SC 기반 검색 단계만 건너뛴다.
- class_only fallback이 prior candidates를 회수하면 item-level 파싱 후 overlap 평가를 계속한다.
- 빈 primary_codes로 인해 `return []` 또는 score 고정 로직을 넣으면 안 된다.
